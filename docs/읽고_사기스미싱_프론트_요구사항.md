## 1. 핵심 결정

**목데이터로 구현하지 않는다.**

홈의 "스미싱 예시"는 목 JSON을 여는 기능이 아니라, 데모용 스미싱 캡처 이미지를 실서버 `POST /analyses`로 보내고 백엔드가 실제 분석한 결과를 보여주는 기능이다.

프론트의 책임은 다음 3개다.

1. 스미싱 데모 이미지를 실서버 분석으로 보낸다.
2. 분석 결과의 `risk.type`에 따라 위험별 도움 CTA를 보여준다.
3. 결과 공유 시 "도움 요청" 템플릿을 생성한다.

## 2. 사용자 플로우

### 2.1 스미싱 예시 분석

```text
홈
  → 예시로 체험하기
  → 스미싱 문자
  → 데모 이미지 로드
  → POST /analyses
  → 분석 중 화면
  → 결과 화면
      → 빨강/노랑 위험
      → 할 일
      → 위험 상세
      → 118 도움 CTA
```

### 2.2 위험별 도움 CTA

```text
결과 화면
  → 위험 카드 탭
  → 위험 상세
      → 백엔드 steps/detail 표시
      → 추천 도움처 CTA
      → 전화 앱 열기
      → 도움받을 곳 전체 보기
```

### 2.3 도움 요청 공유

```text
결과 화면
  → 공유 버튼
  → 결과 공유 / 도움 요청 공유 선택
  → 도움 요청 공유 선택
  → 시스템 공유 시트
```

## 3. 화면 요구사항

### FE-01. 홈 예시 목록에 "스미싱 문자" 추가

| 항목 | 요구사항 |
|---|---|
| 위치 | `HomeView`의 "예시로 체험하기" 바텀시트 |
| 라벨 | `스미싱 문자` |
| 서브텍스트 | `사기 위험 · 118 도움` |
| 아이콘 | `Icons.sms_failed_outlined` 또는 유사한 경고/문자 아이콘 |
| 동작 | 번들 데모 이미지를 읽어 실서버 분석으로 보냄 |

#### 구현 규칙

- 목 JSON을 열지 않는다.
- `AnalyzeController.openSample(id)`처럼 목 결과를 바로 열지 않는다.
- 실제 이미지 bytes를 만들어 기존 `AnalyzeController.analyze(...)` 경로를 탄다.
- `output_mode`와 `lang`은 현재 `ProfileService` 설정을 따른다. 단, 스미싱 예시는 쉬운 한국어 데모가 가장 선명하므로 발표 기기에서는 `easy_korean/ko`를 권장한다.

#### 데모 이미지 자산

권장 위치:

```text
assets/demo/smishing_nhis_ko.png
```

이미지 내용 예:

```text
[국민건강보험]
미납 보험료 68,000원
오늘까지 납부하지 않으면 압류됩니다.
확인: http://nhis-pay-check.example
```

보안 규칙:

- 실제 악성 URL을 넣지 않는다.
- `.example` 도메인을 쓴다.
- 실제 이름, 전화번호, 계좌번호를 넣지 않는다.

### FE-02. 스미싱 분석 요청

기존 `AnalysisRepository.analyze(...)` 흐름을 그대로 쓴다.

요청 형태:

```json
{
  "image_base64": "<demo image base64>",
  "media_type": "image/png",
  "output_mode": "easy_korean",
  "lang": "ko",
  "save": true
}
```

#### 상태 처리

| 상태 | UI |
|---|---|
| 요청 중 | 기존 분석 로딩 화면 |
| 성공 | 기존 결과 화면 |
| 400/413 | "이미지를 다시 선택해 주세요" |
| 422 | "분석에 실패했어요. 다시 시도해 주세요" |
| 429 | "잠시 후 다시 시도해 주세요" |
| 500/502/네트워크 | 연결 실패 화면 또는 기존 폴백 정책 |

주의:

- 이번 요구사항은 목데이터 기능이 아니므로, 스미싱 예시 버튼 자체가 목 결과를 보장하면 안 된다.
- 단, 앱 전체의 기존 라이브 실패 폴백 정책은 유지해도 된다. 폴백을 쓰더라도 화면에 "예시 결과" 표시가 있어야 한다.

### FE-03. 위험별 도움 CTA 표시

`RiskDetailView` 하단에 위험 종류별 추천 도움처를 표시한다.

#### 입력

기존 위험 상세 arguments:

```dart
{
  'level': risk.level,
  'type': risk.type,
  'message': risk.message,
  'detail': risk.detail,
  'steps': risk.steps,
}
```

#### API 호출

위험 상세 진입 시 다음 API를 호출한다.

```http
GET /help-contacts?lang={profile.lang}&doc_type={analysis.doc_type}&risk_type={risk.type}
```

현재 Retrofit client에는 `risk_type` 파라미터가 없으므로 프론트 변경이 필요하다.

```dart
@GET('/help-contacts')
Future<HelpContactsResponse> helpContacts(
  @Query('lang') String? lang,
  @Query('doc_type') String? docType,
  @Query('risk_type') String? riskType,
);
```

#### Fallback 매핑

API 실패 시 로컬 매핑을 사용한다.

| risk.type 포함 문자열 | 우선 연락처 | 보조 연락처 |
|---|---|---|
| `사기`, `스미싱`, `피싱`, `보이스피싱` | 118 KISA/보호나라 | 112 경찰 |
| `임금`, `급여`, `일당`, `부당공제` | 1350 고용노동부 | 1345 외국인종합안내센터 |
| `계약`, `숙소비`, `공제` | 1350 고용노동부 | 1345 외국인종합안내센터 |
| `건강보험` | 1577-1000 국민건강보험공단 | 도움받을 곳 전체 |
| `과태료`, `기한`, 기타 | 도움받을 곳 전체 | 없음 |

#### UI 요구

위험 상세 하단에 아래 블록을 추가한다.

```text
도움받을 곳
혼자 판단하지 않아도 돼요. 아래 기관에 확인해보세요.

[118 전화] KISA/보호나라 · 스미싱·해킹 상담
[112 전화] 경찰 · 긴급 범죄 신고
[도움받을 곳 전체 보기]
```

#### 전화 동작

- CTA 탭 시 `tel:` URL로 전화 앱을 연다.
- 앱이 바로 통화를 시작하지 않고 OS 전화 확인 화면으로 넘긴다.
- 전화 앱 실행 실패 시 스낵바를 보여준다.

```text
전화 앱을 열 수 없어요. 번호를 직접 눌러주세요: 118
```

### FE-04. 도움 요청 공유 템플릿

결과 화면 공유 버튼에서 공유 타입을 선택할 수 있어야 한다.

선택지:

1. 결과 공유
2. 도움 요청으로 공유

시간이 부족하면 "도움 요청으로 공유"를 기본 공유 문구로 사용해도 된다.

#### 도움 요청 공유 텍스트

```text
[읽고] 제가 받은 글을 쉽게 정리했어요.

요약: {summary_one_line}
위험: {risk_label} · {risk_type} — {risk_message}
해야 할 일:
1. {action_1}
2. {action_2}

도움받을 곳: {contact_title} {phone}

혹시 같이 확인해줄 수 있나요?
```

#### 필드 선택 규칙

| 항목 | 규칙 |
|---|---|
| 요약 | `summary_one_line`, 없으면 `doc_type` 기반 기본 문구 |
| 위험 | 가장 높은 위험 1개. `red > yellow > green` |
| 할 일 | priority 낮은 순서 1~2개 |
| 연락처 | `risk.type` 기반 추천 1순위 |
| 원문 | 기본 포함하지 않음 |
| 이미지 | 포함하지 않음 |

#### 개인정보 규칙

- `original_text` 전체를 공유하지 않는다.
- 이미지 파일을 공유하지 않는다.
- 계좌번호, 주민번호, 전화번호처럼 보이는 긴 숫자는 템플릿에 직접 넣지 않는다.
- 금액/기한은 `cards.amount`, `cards.deadline` 수준만 허용한다.

## 4. 데이터 매핑

### 위험 심각도 정렬

```text
red = 3
yellow = 2
green = 1
unknown = 2
```

알 수 없는 level은 yellow처럼 취급한다. 위험을 숨기지 않는다.

### 위험 라벨

| level | 라벨 |
|---|---|
| red | 빨강 |
| yellow | 노랑 |
| green | 초록 |
| 기타 | 주의 |

### 추천 연락처 모델

프론트 내부에서 아래 형태로 다뤄도 된다.

```dart
class RecommendedContact {
  final String title;
  final String org;
  final String phone;
  final String reason;
  final bool emergency;
}
```

서버 응답 `HelpContact`를 그대로 써도 된다. 단, `reason` 문구는 프론트 로컬 카피로 보강한다.

## 5. 수용 기준

### AC-FE-01. 라이브 스미싱 예시

- [ ] 홈 예시 바텀시트에 "스미싱 문자"가 보인다.
- [ ] 탭하면 목 결과가 아니라 실서버 분석 요청이 발생한다.
- [ ] 분석 중 화면을 거친다.
- [ ] 성공 시 결과 화면으로 이동한다.
- [ ] 결과에 사기/스미싱 위험이 보인다.

### AC-FE-02. 위험별 도움 CTA

- [ ] 사기/스미싱 위험 상세에서 118 CTA가 보인다.
- [ ] 임금/계약 위험 상세에서 1350 CTA가 보인다.
- [ ] API 실패 시에도 fallback 연락처가 보인다.
- [ ] CTA 탭 시 전화 앱이 열린다.
- [ ] 도움받을 곳 전체 보기 버튼은 유지된다.

### AC-FE-03. 도움 요청 공유

- [ ] 공유 버튼에서 도움 요청 공유를 실행할 수 있다.
- [ ] 텍스트에 요약, 위험, 할 일, 추천 연락처가 들어간다.
- [ ] 스미싱 결과 공유에는 118이 들어간다.
- [ ] 임금 결과 공유에는 1350이 들어간다.
- [ ] 원문 전체와 이미지는 공유되지 않는다.

## 6. 테스트 요구사항

### Unit

- 위험 타입 `사기` → 118 추천.
- 위험 타입 `스미싱` → 118 추천.
- 위험 타입 `임금` → 1350 추천.
- 위험 타입 `계약` → 1350 추천.
- 알 수 없는 위험 → fallback 도움받을 곳.
- 도움 요청 공유 텍스트에 원문 전체가 들어가지 않음.

### Widget / Flow

- 홈 예시 바텀시트에 스미싱 옵션 표시.
- 위험 상세에 추천 도움처 섹션 표시.
- 공유 선택지가 표시되거나 도움 요청 템플릿이 기본 공유로 실행됨.

### Manual

- 실서버 연결 상태에서 스미싱 예시 분석 성공.
- 네트워크 오류 시 앱이 크래시하지 않음.
- iOS/Android에서 `tel:` 링크 동작 확인.
- 공유 시트가 정상으로 뜨는지 확인.
