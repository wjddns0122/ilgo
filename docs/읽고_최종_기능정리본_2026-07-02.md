## 1. 한 줄 결론

**읽고는 이제 "사진을 읽는 앱"이 아니라 "한국 생활문서를 이해하고 바로 행동하게 해주는 앱"으로 밀어야 한다.**

Google, Apple, Papago, Seeing AI 계열은 이미 사진 속 텍스트 인식과 번역을 잘한다. 그래서 남은 시간은 OCR/번역 기능을 늘리는 데 쓰면 안 된다. 읽고의 차별점은 한국 생활 맥락에서 **위험 신호등, 기한, 할 일, 답장, 도움받을 곳**까지 이어지는 것이다.

최종 전략은 다음과 같다.

1. 이미 구현된 핵심 플로우는 잠근다.
2. 추가 기능은 **사기/스미싱 샘플**, **위험별 도움 CTA**, **도움 요청 공유**만 우선한다.
3. 나머지는 과감히 자른다.

### P1. 시간이 있으면 지금 더 넣을 것

#### P1-1. 사기/스미싱 전용 샘플 추가

**결정:** 추가 추천.

현재 샘플은 건강보험 고지서, 주차 과태료, 사장님 문자다. PRD는 사기/임금/기한을 위험 축으로 잡았는데, 사기/스미싱이 전용 데모로는 비어 있다.

**사용자 장면**

민수가 문자 캡처를 넣는다.

> `[국민건강보험] 미납 보험료 확인: http://...`

읽고가 이렇게 보여준다.

- 빨강: 사기 의심
- 이유: 링크로 돈/개인정보를 요구할 수 있음
- 할 일: 링크 누르지 않기, 118에 문의하기, 공식 앱/전화로 확인하기
- 도움: KISA 118

**구현 모양**

| 항목 | 내용 |
|---|---|
| 새 자산 | `assets/mock/analyze_easy_smishing.json` |
| 목록 추가 | `assets/mock/list.json`에 `anl_0004` |
| 레포지토리 | `MockAnalysisRepository.getById` 매핑 추가 |
| 홈 UI | 샘플 바텀시트에 "스미싱 문자" 추가 |
| 데이터 계약 | 변경 없음. 기존 `risks.type = "사기"` 사용 |
| 검증 | 샘플 선택 → 결과 → 위험 상세 → 도움 CTA |

**왜 높은 우선순위인가**

- KISA 118과 연결되면 공신력 있는 사회안전 기능이 된다.
- "AI가 위험을 잡아준다"가 가장 직관적으로 보인다.
- 새 백엔드가 필요 없다.

#### P1-2. 위험 종류별 도움 CTA

현재 `RiskDetailView`에는 "도움받을 곳 보기" 버튼이 있지만, 위험 종류별로 가장 맞는 기관을 바로 보여주지는 않는다. 이미 `HelpView`와 전화 연결 코드는 있으니, 새 화면보다 **위험 상세 하단에 추천 연락처 1-2개를 보여주는 방식**이 좋다.

#### P1-3. 도움 요청 공유 템플릿

**결정:** 추가 추천.

결과 공유는 이미 있지만 일반 공유 텍스트다. 사회가치 서사를 더 잘 보이게 하려면 공유 메뉴에 **"도움 요청으로 공유"**를 하나 더 둔다.

**공유 문구 예시**

```text
[읽고] 제가 받은 글을 쉽게 정리했어요.

요약: 내일 새벽 5시까지 오라고 하고, 늦으면 일당을 깎겠다고 해요.
위험: 임금 문제일 수 있어요.
할 일: 근로계약서에서 임금·숙소비 공제 내용을 확인해요.

혹시 같이 확인해줄 수 있나요?
```

**왜 좋은가**

- 문제 정의였던 "매번 가족/동료에게 물어봐야 하는 의존"을 "내가 정리해서 도움을 요청하는 연결"로 바꾼다.
- 발표 때 감정적으로 잘 먹힌다.
- 구현은 `SharePlus` 텍스트 분기라 작다.

#### P1-4. 데모 모드 표식

**결정:** 앱 내부 기능보다는 발표 안정성 작업으로 추천.

이미 `USE_MOCK=true` 빌드가 가능하고, 샘플/폴백 배너도 있다. 데모 당일에는 아래 2개 빌드를 모두 준비한다.

| 빌드 | 용도 |
|---|---|
| 실서버 빌드 | 라이브 AI 시연 |
| 목/오프라인 빌드 | 와이파이/Railway 실패 시 즉시 전환 |

목 빌드에는 화면 상단 어딘가에 작게 "예시 결과"가 보이면 충분하다. 심사위원을 속이면 안 된다. 샘플은 샘플이라고 말하는 편이 신뢰를 만든다.

## 6. P2. 남으면 해도 되는 것

| 기능 | 판단 | 이유 |
|---|---|---|
| 촬영 품질 팁 | 선택 | 흔들림/어두움/작은 글씨 실패를 줄인다. 다만 현재 샘플 데모가 있으면 발표 리스크는 낮다. |
| 위험 판단 근거 하이라이트 | 선택 | "왜 빨강인가" 신뢰를 높인다. 단, 제대로 하려면 `risk.evidence` 같은 스키마가 필요하다. 지금은 `원문 보기`로 충분할 수 있다. |
| 도움받을 곳 API 정교화 | 선택 | 이미 폴백이 좋다. 백엔드가 안정적이면 나중에 언어/위험별 추천으로 확장. |
| 샘플 이미지 썸네일 | 선택 | 바텀시트가 더 실제 같아진다. 없어도 기능 가치는 전달된다. |
| 결과 PDF/이미지 저장 | 보류 | 개인정보/저장 정책 설명이 필요해진다. 해커톤 제출 전에는 무겁다. |

| 새 백엔드 대형 API | 프론트 로컬 매핑으로 가능한 것부터 끝낸다. |

## 9. 발표 데모 구성

### 1분 핵심 데모

```text
홈
  → 예시로 체험하기
  → 사장님 문자

  → 할 일 체크
  → 답장 탭에서 "정중하게 다시"
  → 도움받을 곳 1350
```

### 2분 확장 데모

```text
홈
  → 예시로 체험하기
  → 스미싱 문자
  → 사기 위험 빨강
  → "링크 누르지 않기" 할 일 체크
  → KISA 118 도움 CTA
  → 도움 요청 공유
```

### 라이브 AI 데모

```text
종이 찍기 또는 화면 캡처 넣기
  → 실서버 분석
  → 실패하면 폴백 배너가 뜬다는 점까지 신뢰 요소로 설명
```

발표 멘트:

> 라이브 AI가 실패해도 사용자는 빈 화면을 보지 않습니다. 읽고는 위험한 글을 만나는 순간을 위해 만든 앱이라, 실패도 안전하게 처리합니다.
