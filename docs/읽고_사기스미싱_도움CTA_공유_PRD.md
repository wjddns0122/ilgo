## 1. TL;DR

이번 추가 범위는 3개다.

1. **사기/스미싱 샘플 추가**
2. **위험별 도움 CTA**
3. **도움 요청 공유 템플릿**

목표는 기능 개수를 늘리는 것이 아니라, 읽고의 차별점인 **"위험을 발견하고 실제 도움 요청까지 이어주는 앱"**을 데모에서 직접 보여주는 것이다.

대기업 번역/OCR 앱은 사진 속 글자를 읽고 번역한다. 읽고는 여기서 한 걸음 더 가서, 사용자가 손해 보기 전에 **무엇을 누르지 말아야 하는지, 어디에 확인해야 하는지, 누구에게 도움을 요청해야 하는지**까지 알려준다.

---

## 2. 문제 정의

현재 앱은 건강보험 고지서, 과태료, 임금/계약 위험 샘플을 통해 "기한"과 "임금" 축은 잘 보여준다. 하지만 PRD의 핵심 위험 축 중 하나인 **사기/스미싱**은 데모에서 직접 보이지 않는다.

해커톤 심사위원이 이해해야 하는 포인트는 단순하다.

> 한국어를 읽기 어려운 사람이 스미싱 문자를 받으면 링크를 눌러도 되는지 판단하기 어렵다. 읽고는 사진 한 장으로 위험을 빨강으로 알려주고, 링크를 누르지 말라고 말하고, 118 같은 공식 도움처로 연결한다.

이 장면은 사회적 가치, AI 활용, 발표 설득력을 동시에 올린다.

## 5. 최종 사용자 플로우

### 5.1 사기/스미싱 샘플 플로우

```text
홈
  → 예시로 체험하기
  → 스미싱 문자
  → 결과 화면
      → 빨강: 사기/스미싱 의심
      → 할 일: 링크 누르지 않기, 공식 고객센터/118에 확인
      → 위험 상세
          → 118 상담 CTA
          → 경찰 112 보조 CTA
      → 도움 요청 공유
```

### 5.2 임금 위험 도움 CTA 플로우

```text
홈
  → 예시로 체험하기
  → 사장님 문자
  → 결과 화면
      → 빨강: 임금 위험
      → 위험 상세
          → 고용노동부 1350 CTA
          → 외국인종합안내센터 1345 보조 CTA
```

### 5.3 도움 요청 공유 플로우

```text
결과 화면
  → 공유 버튼
  → 도움 요청으로 공유
  → 시스템 공유 시트
  → 가족/동료/지원기관에게 전송
```

### FR-HELP-01. 위험별 도움 CTA

| 항목 | 내용 |
|---|---|
| 우선순위 | P1 |
| 사용자 가치 | 위험 신호등이 실제 도움 요청으로 이어진다. |
| 프론트 | `RiskDetailView` 하단에 위험 타입별 추천 연락처 표시 |
| 백엔드 | `/v1/help-contacts`가 `risk_type` 필터를 받을 수 있게 확장 |
| 오프라인 | 프론트 로컬 fallback 매핑 유지 |

#### CTA 매핑

| 위험 타입 | 1순위 | 2순위 | 카피 |
|---|---|---|---|
| 사기, 스미싱 | KISA/보호나라 118 | 경찰 112 | "링크를 누르기 전에 118에 확인해보세요." |
| 임금, 계약, 부당공제 | 고용노동부 1350 | 외국인종합안내센터 1345 | "임금 문제는 1350에서 상담할 수 있어요." |
| 과태료, 기한 | 문서 속 문의번호 | 도움받을 곳 전체 | "기한 안에 공식 기관에 확인하세요." |
| 건강보험, 공공요금 | 국민건강보험공단 1577-1000 등 | 도움받을 곳 전체 | "공식 고객센터로 직접 확인하세요." |
| 기타 | 도움받을 곳 전체 | 112(긴급 시) | "혼자 판단하기 어렵다면 상담을 받아보세요." |

#### Acceptance Criteria

- 사기/스미싱 위험 상세에는 118 CTA가 먼저 보인다.
- 임금/계약 위험 상세에는 1350 CTA가 먼저 보인다.
- CTA를 누르면 전화 앱이 열린다. 앱이 바로 전화를 걸지는 않고, 사용자가 한 번 더 확인한다.
- 네트워크가 없어도 기본 fallback 연락처가 보인다.
- "도움받을 곳 보기" 전체 화면은 계속 접근 가능하다.

### FR-SHARE-01. 도움 요청 공유 템플릿

| 항목 | 내용 |
|---|---|
| 우선순위 | P1 |
| 사용자 가치 | 가족/동료에게 "이거 봐줘"가 아니라 정리된 도움 요청을 보낼 수 있다. |
| 프론트 | 결과 공유 메뉴에 "도움 요청으로 공유" 추가 |
| 백엔드 | 별도 변경 없음 |
| 개인정보 | 원문 전체는 기본 공유하지 않는다. 요약, 위험, 할 일, 도움처만 공유한다. |

#### 공유 템플릿

```text
[읽고] 제가 받은 글을 쉽게 정리했어요.

요약: {summary_one_line}
위험: {top_risk_level} · {top_risk_type} — {top_risk_message}
해야 할 일:
1. {action_1}
2. {action_2}

도움받을 곳: {recommended_contact_title} {phone}

혹시 같이 확인해줄 수 있나요?
```

#### 사기/스미싱 예시 출력

```text
[읽고] 제가 받은 글을 쉽게 정리했어요.

요약: 건강보험 미납 안내처럼 보이지만 사기 문자일 수 있어요.
위험: 빨강 · 사기 — 공식 문자인지 확인되지 않은 링크가 있어요. 스미싱일 수 있으니 누르지 마세요.
해야 할 일:
1. 문자 안의 링크를 누르지 마세요.
2. 국민건강보험공단 공식 앱이나 고객센터로 직접 확인하세요.

도움받을 곳: KISA/보호나라 118

혹시 같이 확인해줄 수 있나요?
```

#### Acceptance Criteria

- 공유 텍스트에 요약, 가장 높은 위험, 할 일 1~2개가 들어간다.
- 사기/스미싱이면 118이 포함된다.
- 임금/계약이면 1350이 포함된다.
- 원문 전체와 이미지 원본은 기본 공유하지 않는다.
- 샘플/폴백 결과에서도 공유가 가능하다.

## 7. 백엔드 요구사항

### 7.1 `POST /v1/analyses` 스미싱 분석 요구

기존 응답 스키마는 유지한다. 단, AI 프롬프트와 후처리에서 스미싱 메시지를 명확히 다룬다.

#### 입력 예시

```text
[국민건강보험] 미납 보험료 68,000원. 오늘까지 납부하지 않으면 압류됩니다. 확인: http://nhis-pay-check.example
```

#### 출력 규칙

| 필드 | 요구 |
|---|---|
| `doc_type` | `메시지` |
| `risks[].level` | 스미싱 의심 링크가 있으면 `red`, 확신이 약하면 `yellow` |
| `risks[].type` | MVP 권장값은 `사기`. 백엔드가 별도 분류를 원하면 `스미싱`도 허용 |
| `risks[].message` | 링크를 누르지 말고 공식 채널로 확인하라는 내용 |
| `actions[]` | 링크 미클릭, 공식 채널 확인, 118 상담 |
| `cards.amount` | 본문 금액이 있으면 정확히 추출 |
| `cards.deadline` | 본문에 실제 날짜가 없고 "오늘"만 있으면 null 권장 |
| `reply_drafts` | easy_korean 모드에서는 빈 배열 |

#### 프롬프트 추가 문구

```text
스미싱/피싱/사기 의심 문자는 별도 위험으로 판단한다.
공식 기관명을 사칭하면서 낯선 URL, 단축 URL, 압박 문구(오늘까지, 압류, 즉시 납부),
개인정보/금전 입력 유도, 앱 설치 유도가 있으면 red 또는 yellow 위험으로 표시한다.
확신할 수 없으면 "사기입니다"라고 단정하지 말고 "사기/스미싱이 의심돼요"라고 말한다.
행동에는 링크를 누르지 않기, 공식 앱/대표번호로 직접 확인하기, 118 상담을 포함한다.
```

### 7.2 `GET /v1/help-contacts` 확장

현재 프론트 클라이언트는 다음 엔드포인트를 가진다.

```http
GET /v1/help-contacts?lang=ko&doc_type=메시지
```

이번 범위에서는 선택 쿼리를 추가한다.

```http
GET /v1/help-contacts?lang=ko&doc_type=메시지&risk_type=사기
GET /v1/help-contacts?lang=en&risk_type=임금
```

#### 요청 쿼리

| 파라미터 | 필수 | 설명 |
|---|---|---|
| `lang` | no | 사용자 언어. `ko`, `en` 등 |
| `doc_type` | no | `메시지`, `고지서`, `계약` 등 |
| `risk_type` | no | `사기`, `스미싱`, `임금`, `계약`, `과태료`, `기한`, `기타` |

#### 응답 스키마

기존 `HelpContactsResponse` 유지.

```json
{
  "items": [
    {
      "id": "kisa_118",
      "title": "스미싱·해킹 상담",
      "org": "KISA/보호나라",
      "phone": "118",
      "category": "사기",
      "emergency": false
    },
    {
      "id": "police_112",
      "title": "긴급 · 범죄 신고",
      "org": "경찰",
      "phone": "112",
      "category": "긴급",
      "emergency": true
    }
  ]
}
```

#### 백엔드 구현 옵션

| 옵션 | 설명 | 추천 |
|---|---|---|
| Static config | Java/Kotlin 상수 또는 YAML로 연락처/매핑 관리 | 해커톤 MVP 추천 |
| DB table | `help_contacts` 테이블로 관리 | 운영 확장 시 |

MVP는 static config로 충분하다. 연락처 데이터는 작고, 발표 전 운영툴이 필요 없다.

### 7.3 연락처 기본 데이터

| id | title | org | phone | category | emergency |
|---|---|---|---|---|---|
| `kisa_118` | 스미싱·해킹 상담 | KISA/보호나라 | 118 | 사기 | false |
| `moel_1350` | 임금·노동 상담 | 고용노동부 | 1350 | 임금 | false |
| `immigration_1345` | 외국인종합안내센터 | 법무부 | 1345 | 외국인 | false |
| `nhis_1577_1000` | 건강보험 문의 | 국민건강보험공단 | 1577-1000 | 건강보험 | false |
| `police_112` | 긴급 · 범죄 신고 | 경찰 | 112 | 긴급 | true |

공식 기준:

- KISA/보호나라: 스미싱 확인서비스, 해킹/스팸/개인정보침해 상담 `118`
- 고용노동부: 노동 상담 `1350`

### 7.4 DB 변경

MVP에서는 DB 변경을 요구하지 않는다.

이유:

- `risks.type`은 현재 varchar 계열로 설계되어 있다.
- `HelpContact` 응답 스키마에 `category`가 이미 있다.
- 도움처는 static config로 충분하다.

단, 백엔드가 `risk.type`을 CHECK 제약이나 enum으로 강하게 제한했다면 아래 값을 허용해야 한다.

```text
사기, 스미싱, 임금, 계약, 부당공제, 기한, 과태료, 건강보험, 기타
```

## 8. 프론트 요구사항

### 8.1 홈 샘플 옵션

현재 홈 예시 바텀시트에 샘플 3종이 있다.

- 건강보험 고지서
- 주차 과태료 고지서
- 사장님 문자

여기에 아래 옵션을 추가한다.

| 필드 | 값 |
|---|---|
| title | 스미싱 문자 |
| subtitle | 사기 위험 · 118 도움 |
| icon | `Icons.sms_failed_outlined` 또는 유사 아이콘 |
| id | `anl_0004` |

### 8.2 위험 상세 CTA

`RiskDetailView`에서 `level`, `type`, `message`, `steps`를 받고 있다. 여기에 `type` 기반 추천 연락처를 렌더링한다.

```text
위험 상세
  body
  steps
  추천 도움처
    [118 전화] [도움받을 곳 보기]
```

구현은 두 단계로 한다.

1. 로컬 fallback 매핑으로 먼저 표시한다.
2. 백엔드 `/help-contacts?risk_type=...`가 준비되면 API 결과를 우선한다.

### 8.3 도움 요청 공유

현재 결과 헤더 공유 버튼은 일반 결과 공유를 실행한다. 이번 범위에서는 공유 버튼 탭 시 간단한 선택지를 제공한다.

```text
공유
  1. 결과 공유
  2. 도움 요청으로 공유
```

선택지가 UX상 무겁다면, 우선은 "도움 요청 공유"를 기본 공유 텍스트로 바꿔도 된다. 해커톤 데모에서는 도움 요청 템플릿이 더 강하다.

### 8.4 개인정보 기본값

도움 요청 공유는 원문 전체를 공유하지 않는다.

공유 포함:

- 요약
- 위험 1개
- 할 일 1~2개
- 추천 도움처

공유 제외:

- 원문 전체
- 이미지
- 계좌번호/주민번호/전화번호 원문

## 11. 에러/엣지 케이스

| 케이스 | 기대 동작 |
|---|---|
| `risks`가 비어 있음 | 도움 CTA 숨김, 도움받을 곳 전체 버튼만 표시 |
| `risk.type`이 알 수 없는 문자열 | 도움받을 곳 fallback 목록 표시 |
| 전화 앱 실행 실패 | 스낵바: "전화 앱을 열 수 없어요. 번호를 직접 눌러주세요." |
| 백엔드 `/help-contacts` 실패 | 로컬 fallback 연락처 표시 |
| 공유 앱 없음 | 플랫폼 기본 에러 처리, 앱 크래시 없음 |
| 샘플 JSON 누락 | 스낵바: "예시를 불러오지 못했어요." |
| AI가 사기를 green으로 반환 | 프롬프트/eval fixture로 회귀 테스트. 확신 부족이면 최소 yellow 권장 |
