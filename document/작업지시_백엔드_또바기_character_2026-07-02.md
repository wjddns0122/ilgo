# 읽고 — 백엔드 작업지시: 또바기 캐릭터 상태(AI 판단) 추가

| 항목 | 내용 |
|---|---|
| 작성일 | 2026-07-02 |
| 대상 | Spring Boot 백엔드 (분석 프롬프트 담당) |
| 범위 | `POST /v1/analyses` 응답에 **옵셔널 필드 2개 추가**. 추가 API 호출 없음(기존 1회 멀티모달 호출 JSON에 필드만 더함). |
| 프론트 상태 | **이미 대응 완료** — `character_state`/`character_line` 있으면 쓰고, 없거나 이상하면 위험도로 자동 폴백. 안 줘도 앱 정상. |

> **한 줄:** AI가 문서를 분석하면서 "이 글엔 또바기 어떤 표정이 맞나"까지 골라 응답 JSON에 넣어준다. 프론트는 그 표정·대사를 결과화면 또바기에 그대로 뿌린다.

---

## 1. 추가 필드 (기존 `ReadResult` 스키마에 옵셔널 2개)

| 필드 | 타입 | 필수 | 설명 |
|---|---|---|---|
| `character_state` | string(enum) | no | 또바기 표정. `warn` \| `caution` \| `safe` \| `victim` 중 하나 |
| `character_line` | string | no | 또바기 한마디(≤ 40자). 출력 언어(`lang`)로. 아래 톤 규칙 |

- 둘 다 **옵셔널**. 못 정하겠으면 아예 빼도 됨(프론트가 위험도로 폴백).
- 기존 필드/스키마는 그대로. DB 변경 불필요(`character_*`는 응답에만 실어도 됨, 저장은 선택).

### `character_state` 선택 기준
| 값 | 언제 | 프론트 아트 |
|---|---|---|
| `warn` | 사기/스미싱/긴급 위험 — 사칭·낯선 링크·압박·금전/개인정보 요구 | 놀란 또바기(빨강) |
| `victim` | **이미 피해 정황** — 본문에 "돈 보냄/송금/링크 눌렀다/앱 설치함" 등 당한 흔적, 또는 매우 심각 | 우는 또바기(빨강) |
| `caution` | 확인 필요 — 애매한 위험, 기한 임박, 부당공제 의심 | 갸웃 또바기(노랑) |
| `safe` | 안전/일반 안내 — 위험 신호 없음 | 안심 또바기(초록) |

> 확신 없으면 `warn`보다 `caution`(과경보 방지), 단 사기 신호가 뚜렷하면 `warn`.

### `character_line` 톤 규칙
- 짧게(≤40자), 출력 언어(`lang`)로.
- 한국어: **`safe`=다정한 반말**("이건 안전해 보여, 걱정 마"), **`warn`/`caution`/`victim`=존댓말**("이거 위험할 수 있어요, 링크 누르지 마세요").
- 영어 등 기타 언어: 반말/존댓말 구분 없이 **따뜻하고 명확한 단일 톤**.
- **금지:** "보상해준다/돌려준다" 단정. 대신 "확인해요 / 신고해요 / 공식 절차가 있어요" 안내 톤.

## 2. 분석 시스템 프롬프트에 추가할 문구 (그대로 삽입)

```text
[또바기 캐릭터 필드]
분석 결과에 아래 두 필드를 추가한다. 확신이 없으면 생략해도 된다.

character_state: 이 문서에 어울리는 캐릭터 표정을 하나 고른다.
  - "warn"    : 사기·스미싱·긴급 위험(기관 사칭, 낯선/단축 링크, "오늘까지·압류·즉시" 압박,
                금전·개인정보·인증번호 요구, 앱 설치 유도)이 뚜렷할 때.
  - "victim"  : 사용자가 이미 당한 정황(돈을 보냄/송금/이체, 링크를 누름, 앱을 설치함,
                개인정보를 입력함 등)이 본문에 있거나 피해가 임박·심각할 때.
  - "caution" : 위험이 애매하거나 기한 임박·부당공제 의심 등 확인이 필요할 때.
  - "safe"    : 위험 신호가 없고 일반적인 안내·안전한 문서일 때.
  확신이 없으면 warn보다 caution을 택한다. 단 사기 신호가 명확하면 warn.

character_line: 또바기가 사용자에게 건네는 짧은 한마디(40자 이내), 출력 언어로 작성.
  - 한국어: safe는 다정한 반말, warn/caution/victim은 존댓말.
  - 그 외 언어: 반말/존댓말 구분 없이 따뜻하고 명확한 단일 톤.
  - "보상/환급을 보장한다"고 단정하지 말고, "확인하세요 / 신고하세요 / 공식 절차가 있어요"처럼 안내한다.
  - 링크가 있으면 "누르지 마세요"를 우선한다.
```

## 3. 응답 예시 (스미싱 — warn)

```json
{
  "id": "server-id",
  "status": "done",
  "output_mode": "easy_korean",
  "lang": "ko",
  "doc_type": "메시지",
  "character_state": "warn",
  "character_line": "이 링크 누르지 마세요. 공식 앱으로 확인해요.",
  "summary_one_line": "건강보험 미납 안내처럼 보이지만 사기 문자일 수 있어요",
  "risks": [
    { "id": "rsk-1", "level": "red", "type": "사기",
      "message": "공식 문자인지 확인되지 않은 링크가 있어요. 누르지 마세요.",
      "detail": "…", "steps": ["…"] }
  ],
  "actions": [ { "id": "act-1", "text": "문자 안의 링크를 누르지 마세요.", "priority": 1, "is_done": false } ],
  "reply_drafts": [],
  "created_at": "server-time"
}
```

### 피해 정황 예시 (victim)
입력에 "이미 링크 눌러서 앱 깔았어요" 같은 맥락이 있으면:
```json
{ "character_state": "victim", "character_line": "늦지 않았어요. 지금 바로 112에 신고해요." }
```

### 안전 예시 (safe)
```json
{ "character_state": "safe", "character_line": "이건 안전해 보여. 걱정 마." }
```

---

## 4. 수용 기준
- [ ] 스미싱/사기 이미지 → `character_state = warn`(또는 피해 정황 시 victim).
- [ ] 일반 안전 문서 → `character_state = safe`.
- [ ] `character_line`이 출력 언어·톤 규칙을 따른다("보상 보장" 문구 없음).
- [ ] 두 필드를 생략해도 200 응답이 유효하다(옵셔널).
- [ ] 기존 `ReadResult` 스키마·필드는 그대로다.

> 프론트는 유효하지 않은 `character_state`(예: 오타)면 무시하고 위험도로 폴백하므로, 백엔드는 위 4개 값만 정확히 쓰면 된다.
